# Code Review SOP

> **版本**: v3.0
> **最後更新**: 2026-03-22

---

## 概述

Code Review 對應 Gate G2（程式碼審查）和 G3（測試驗收）。所有程式碼修改必須經過 L1 內部 Review，通過後才提交對應 Gate。半成品不得送審。

## 規則

1. 所有程式碼修改必須經過 L1 Review 才能提交 Gate
2. L2 工程師不能自行合併或提交 Gate
3. Review 結果記錄在開發計畫書第 10 節（備查）

## Review 四種類型

| Review 類型 | 檢查對象 | 核心問題 |
|------------|---------|---------|
| **對程式碼** | 程式碼品質、邏輯正確性、coding standards | 有沒有 bug？程式碼品質合格嗎？ |
| **對規範** | 實作 vs `api-design.md`、`data-model.md`、`feature-spec.md` | 實作跟規範文件一致嗎？ |
| **對設計稿** | 實際 UI vs 設計稿 HTML/截圖 | 畫面跟設計稿長一樣嗎？ |
| **對功能** | 端到端功能、效能指標、驗收標準 | 功能跑得通嗎？效能達標嗎？ |

### 步驟 vs Review 類型對應

| 步驟 | 適用 Review 組合 |
|------|-----------------|
| 設計 | ✅ 對規範 |
| UI 圖稿 | ✅ 對設計稿（品牌一致性、響應式） |
| 實作（後端） | ✅ 對程式碼 + ✅ 對規範 |
| 實作（前端） | ✅ 對程式碼 + ✅ 對設計稿 + ✅ 對規範 |
| 測試 | ✅ 對功能 + ✅ 對規範（最終一致性） |
| 文件 | ✅ 對規範（文件與程式碼一致） |

> 以上是通用對應，具體組合在開發計畫書中根據 G0 勾選的步驟調整。

### 設計稿比對清單（「對設計稿」Review 時使用）

| 比對項 | 說明 |
|--------|------|
| 佈局結構 | 元件位置、大小、間距與設計稿一致 |
| 色彩 | 使用設計稿定義的色系，不可自行選色 |
| 動畫 | 時長、easing 與設計稿規格一致 |
| 狀態切換 | 設計稿定義的每個 UI 狀態都要實作並比對 |
| 響應式 | 桌面版和手機版都與設計稿一致 |
| 文案 | 按鈕文字、placeholder、空狀態文案與設計稿一致 |

**L1 比對方式**：同時開啟設計稿 HTML 和實際畫面（或截圖），逐項確認。UI 不一致 = Blocker。

## Review 通過標準

- 0 🔴 Blocker + 0 🟠 Major → **通過**，提交對應 Gate
- 有 Blocker / Major → **不通過**，修正後重新 Review
- 🟡 Minor → 記錄，可在後續步驟或下個 Sprint 處理

## 流程

```
L2 完成任務 → 回報 L1
    ↓
L1 執行內部 Review（選擇適用的 Review 類型組合）
    ↓
通過 → L1 提交 Gate 回報 → PM 審核（6 項 checklist）→ 老闆決策
退回 → L1 附上修改建議 → L2 修正後重新提交
```

### PM 審核 Gate 回報（L1 提交後、老闆審核前）

| # | 檢查項 | 說明 |
|---|--------|------|
| 1 | 交付物完整性 | Gate 要求的所有交付物是否齊全 |
| 2 | 數據正確性 | 報告中的數字是否與實際一致 |
| 3 | 驗收標準對照 | 逐項比對提案書/計畫書的驗收標準 |
| 4 | 流程合規 | 阻斷規則是否遵守、前置 Gate 是否已通過 |
| 5 | 計畫書紀錄 | 開發計畫書第 10 節是否已填寫 |
| 6 | 附帶問題 | 過程中發現的問題是否已記錄 |

PM 審核通過後，整理摘要報告呈報老闆，含 PM 建議（通過/駁回/附條件通過）。

## 與 Gate 的關係

- G2 通過 = Code Review 全部完成，程式碼品質確認
- G3 通過 = 測試驗收完成（單元測試 + E2E 測試 + 效能測試）
- G4 通過 = 文件審查完成（規範文件與程式碼一致）
