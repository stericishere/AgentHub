# Sprint 規劃 SOP

> **版本**: v4.1
> **最後更新**: 2026-03-22

---

## 核心原則

**G0（開發計畫書審核）決定一切後續流程。** 不預設固定模板，每次開發根據計畫書的勾選動態組合步驟和關卡。

---

## 流程概覽

```
老闆 + PM 討論需求
    ↓
撰寫提案書（sprint-proposal.md.template）
    ↓
G0 審核：確認目標、範圍、勾選步驟和關卡
    ↓  ← G0 核心產出：確認的流程（步驟 + 關卡組合）
    ↓
L1 撰寫開發計畫書（dev-plan.md.template）
    ↓  ← 定義每個任務的內容、負責 Agent、驗收標準
    ↓
老闆透過 PM 產生任務指令 → 手動建立任務 → 轉達 Agent
    ↓
執行 → 逐步 Review → 逐關 Gate
    ↓  ← L1 提交 → PM 審核（6 項 checklist）→ 老闆決策 → PM 當下記錄
    ↓
所有勾選關卡通過 → Sprint 完成
```

---

## 步驟與關卡對照

| 步驟 | 說明 | 對應關卡 | 何時需要 |
|------|------|---------|---------|
| 需求分析 | 需求文件、任務拆解 | G0（必選） | 每次都需要 |
| 設計 | 架構/API 設計 | — | 有技術複雜度時 |
| UI 圖稿 | HTML mockup | G1: 圖稿審核 | 有 UI 變更時 |
| 實作 | 程式碼開發 | G2: 程式碼審查 | 有程式碼變更時 |
| 測試 | 單元/E2E 測試 | G3: 測試驗收 | 有程式碼變更時 |
| 文件 | 文件更新 | G4: 文件審查 | 有文件變更時 |
| 部署 | 環境配置、CI/CD | G5: 部署就緒 | 需上線時 |
| 發佈 | 正式對外發佈 | G6: 正式發佈 | 需發佈時 |

> 步驟和關卡的組合不受限。G0 審核時根據實際需求勾選。

### 常見組合範例

| 場景 | 勾選的步驟 | 產生的關卡 |
|------|----------|-----------|
| 全新大功能 | 需求 + 設計 + UI 圖稿 + 實作 + 測試 + 文件 + 部署 | G0→G1→G2→G3→G4→G5 |
| 一般功能 | 需求 + 實作 + 測試 + 文件 | G0→G2→G3→G4 |
| 純後端 | 需求 + 設計 + 實作 + 測試 | G0→G2→G3 |
| 修 Bug | 需求 + 實作 + 測試 | G0→G2→G3 |
| 緊急 Hotfix | 需求 + 實作 | G0→G2 |
| 純文件 | 需求 + 文件 | G0→G4 |
| 上線發佈 | 需求 + 部署 + 發佈 | G0→G5→G6 |
| 自訂 | 任意組合 + 額外步驟 | 依勾選產生 |

## 阻斷規則

阻斷規則在開發計畫書中勾選，常見的：

- **G1 阻斷**：圖稿審核通過前不得開始實作（有 UI 時建議勾選）
- **G3 阻斷**：測試驗收通過前不得進入文件階段
- **自訂阻斷**：計畫書中可自訂任意阻斷規則

---

## 額外步驟

標準步驟之外，計畫書可新增自訂步驟，例如：

- 效能壓測
- 安全掃描
- 第三方 API 整合測試
- 資料遷移驗證
- UX 使用者測試

每個額外步驟可指定審核方式：老闆審核 / L1 自審 / 自動化檢查。

## 任務建立與指令傳達

### 流程

```
開發計畫書定義任務（T1, T2, T3...）
    ↓
老闆透過 PM 產生任務指令
    ↓
老闆手動建立任務，指定負責 Agent
    ↓
透過指揮鏈轉達指令（老闆 → L1 → L2）
    ↓
Agent 執行 → 回報 L1 → L1 Review
    ↓
任務完成紀錄寫入開發計畫書（備查）
```

### 任務定義要求

開發計畫書的任務清單必須包含：
- **任務名稱與說明**：具體描述要做什麼
- **負責 Agent**：指定執行者
- **依賴關係**：標明先後順序
- **驗收標準**：明確完成條件

### 稽核軌跡

開發計畫書第 10 節「任務與審核紀錄」作為 Sprint 完整稽核軌跡：

| 紀錄類型 | 記錄時機 | 記錄內容 |
|---------|---------|---------|
| 任務完成 | 每個任務完成時 | 完成日期、結果、備註 |
| Review 紀錄 | L1 Review 通過時 | Review 文件連結、結果 |
| Gate 紀錄 | Gate 審核完成時 | 決策、審核意見 |

> **原則**：所有任務和審核結果都必須留存在開發計畫書中，確保可追溯。

## Gate 審核流程（三層）

### 流程

```
L1 內部 Review 通過 → L1 提交 Gate 回報
    ↓
PM 審核（6 項 checklist）
    ↓
PM 整理摘要 + 建議（通過/駁回/附條件通過）→ 呈報老闆
    ↓
老闆決策 → PM 當下記錄到計畫書第 10 節
```

### PM 審核 Checklist

L1 提交 Gate 回報後，PM 必須先審核再呈報老闆：

| # | 檢查項 | 說明 |
|---|--------|------|
| 1 | 交付物完整性 | Gate 要求的所有交付物是否齊全（程式碼/測試報告/文件等） |
| 2 | 數據正確性 | 報告中的數字是否與實際一致（通過率、cases 數、覆蓋範圍） |
| 3 | 驗收標準對照 | 逐項比對提案書/計畫書的驗收標準，是否全部達成 |
| 4 | 流程合規 | 阻斷規則是否遵守、前置 Gate 是否已通過 |
| 5 | 計畫書紀錄 | 開發計畫書第 10 節（任務完成/Review/Gate 紀錄）是否已填寫 |
| 6 | 附帶問題 | 過程中發現的問題是否已記錄，是否需要後續處理 |

> **核心邏輯**：L1 提交 → PM 過濾 → PM 摘要+建議 → 老闆決策 → PM 當下記錄。避免半成品直接到老闆面前，也確保稽核軌跡即時留存。

## 內部 Review 機制

L1 領導在**每個勾選的步驟**完成後執行 Review，確保品質合格後才提交對應 Gate。

**原則**：L1 為品質把關，不將半成品送審。

Review checklist 根據勾選的步驟動態產生，見 `templates/internal-review.md.template`。

## 指揮鏈規則

### L1/L2 分工

| 角色 | 職責 | 權限 |
|------|------|------|
| 老闆 | G0 審核開發計畫、Gate 審核 | 決定流程、審核關卡 |
| L1 領導 | 規劃、Review、回報 | 拆解任務、分派 L2、內部 Review |
| L2 執行者 | 執行任務、回報 L1 | 開發、測試、文件撰寫 |

### 禁止事項

- ❌ L2 不得跳過 L1 直接向老闆回報
- ❌ 老闆不得跳過 L1 直接指揮 L2
- ❌ L1 不得在未 Review 的情況下提交 Gate
- ❌ L2 不得擅自修改共用檔案而不通知 L1
